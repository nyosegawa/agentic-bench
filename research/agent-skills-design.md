# Agent Skills 設計: 探索的モデル検証 (2026-02)

## 根本的な発想の転換

### NG: 固定パイプライン型

```
model → smoke.py → quality.py → performance.py → report
```

問題:
- モデルごとに評価コードが違う
- 新しいモデル種別に対応するたびにコード追加が必要
- エラー時の柔軟な対応ができない
- 「出力を見て判断する」ができない

### OK: エージェント探索型

```
model → Agent が model card を読む
      → Agent が適切なコードを書く
      → Agent が実行して出力を見る
      → Agent が判断・評価を言語化する
      → Agent が必要に応じて再試行する
      → Agent がレポートを書く
```

**エージェント自身が ML エンジニアとして振る舞う。**
固定コードではなく、Agent Skill（指示書）が本体。

---

## Coding Agent の能力マップ

### できること（活用すべき）

| 能力 | 具体的にどう使うか |
|------|-----------------|
| **Web 調査** | HF model card、論文、ブログ、GitHub README を読んで理解 |
| **コード生成** | モデルのロード・推論コードをその場で書く |
| **コード実行** | Colab / Modal / beam.cloud 上で実行 |
| **画像認識** | 生成された画像を見て品質を判断（Claude はマルチモーダル） |
| **テキスト評価** | LLM の出力を読んで、品質・正確性・流暢さを評価 |
| **エラー対応** | 実行エラーをデバッグ、依存関係の問題を解決、別アプローチを試行 |
| **比較判断** | 複数モデルの出力を見比べて相対評価 |
| **レポート執筆** | 技術的な評価レポートを自然言語で書く |
| **ツール連携** | Chrome MCP (Colab), SSH, REST API, git |

### できないこと（人間に委ねる部分）

| 制約 | 対応方法 |
|------|---------|
| 音声を聴けない | wav を生成し HTML レポートに `<audio>` 埋め込み。人間が聴いて判断 |
| 動画を見れない | フレーム抽出して画像として評価。全体は `<video>` で人間が確認 |
| 3D モデルを操作できない | three.js ビューアに埋め込み。人間が操作 |
| 主観的な「好み」 | 技術的品質は評価可能。好みは人間判断 |

## Agent Skill の設計思想

### Skill は「指示書」であり「コード」ではない

従来: `run_benchmark(model, provider)` → 固定コード実行
提案: 「このモデルを検証して」→ Agent が状況に応じて判断・実行

### Skill が与えるもの

1. **ゴール**: 何を達成すべきか（モデルの動作確認・品質評価・性能測定）
2. **戦略**: どういう順序で考えるか（まず model card を読む、次に…）
3. **制約**: コスト上限、使えるプロバイダ、結果の保存先
4. **知識**: プロバイダの使い方、よくあるトラブルと対処法
5. **判断基準**: 何を持って「良い」「悪い」とするか

### Skill が与えないもの

- 固定の評価コード（Agent が書く）
- 固定のベンチマークリスト（Agent が選ぶ）
- 固定のプロバイダ選択ロジック（Agent が状況判断する）

## Skill 構成案

### Skill 1: `agentic-bench` (メインスキル)

```markdown
# agentic-bench: 探索的モデル検証

## あなたの役割
あなたは経験豊富な ML エンジニアです。
ユーザーが指定したモデルを、実際に動かして検証してください。

## 検証の進め方

### Step 1: モデルを理解する
- HuggingFace model card / GitHub README / 公式ドキュメントを読む
- モデルの種類、入出力、サイズ、要件を把握する
- 推奨される使い方、サンプルコードを確認する

### Step 2: 実行環境を選ぶ
利用可能なプロバイダと優先順位:
1. HF Inference API（HF Pro 枠）: HF 上のモデルで API 対応なら最優先
2. Google Colab Pro（Chrome MCP）: GPU が必要で ~30B 以下
3. Modal（$30/月無料枠）: Colab で GPU が取れない時、30B+ のモデル
4. beam.cloud: 専用スクリプトが必要な場合

判断基準: モデルの VRAM 要件 vs プロバイダの GPU スペック。
コストは最小に。既に課金済みのサービスを優先。

### Step 3: 動かしてみる
- model card のサンプルコードを参考に、推論コードを書く
- まず最小限の入力で動作確認
- エラーが出たら原因を調べて対処する（依存関係、VRAM不足、形式違い等）

### Step 4: 出力を評価する
- テキスト出力: 読んで品質を評価（流暢さ、正確性、指示遵守）
- 画像出力: 見て品質を評価（構図、ディテール、プロンプト忠実度）
- 音声出力: ファイルとして保存（人間が後で聴いて確認）
- 数値出力: グラフ化して傾向を確認

複数の入力で試して、強み・弱みを把握する。

### Step 5: 性能を測る
- 推論速度（tokens/sec, 秒/枚, RTF 等、モデル種別に応じて）
- 使用した GPU とプロバイダ
- 推定コスト

### Step 6: レポートを書く
結果を `results/YYYY-MM-DD_modelname/` に保存:
- `report.html`: 人間が見るレポート（出力の埋め込み、評価コメント、メトリクス）
- `metrics.json`: 構造化データ（他モデルとの比較用）
- `artifacts/`: 生成物（画像、音声等）

### Step 7: GitHub に commit

## 重要な原則
- **探索的であれ**: 決まったスクリプトを流すのではなく、状況に応じて判断する
- **失敗を恐れるな**: エラーが出たらデバッグして対処する。それも検証結果の一部
- **正直に書け**: 良い点も悪い点も。動かなかったなら「動かなかった」と報告
- **コストを意識**: 特に動画生成や大型モデルはコストが高い。無駄な再実行を避ける
```

## 探索的アプローチの具体例

### 例 1: 新しい LLM を検証

```
ユーザー: "Gemma 3 27B を検証して"

Agent の動き:
1. HF で google/gemma-3-27b の model card を読む
2. 27B → A100 が必要 → Colab Pro で A100 ランタイムを試みる
3. Chrome MCP で Colab を開き、ノートブックを作成
4. model card のサンプルコードを参考にロードコードを書く
5. まずシンプルなプロンプトで動作確認
6. 日本語、英語、コード生成など複数タスクで試す
7. 出力を読んで「日本語は流暢だが、コードは Llama 3 より弱い」等の評価
8. tokens/sec を測定
9. HTML レポートにまとめて commit
```

### 例 2: 未知のモデル種別を検証

```
ユーザー: "TimesFM を検証して"

Agent の動き:
1. google/timesfm で model card を読む → 時系列予測モデルだと理解
2. 必要な依存関係を確認（timesfm パッケージ）
3. CPU で動くか確認 → 動くなら Colab Free / ローカルで十分
4. サンプルデータを用意（sin 波、実際の株価データ等）
5. 予測を実行、結果をグラフ化
6. 「短期予測は正確だが、長期になると精度が落ちる」等の分析
7. matplotlib でグラフを生成、HTML に埋め込み
```

### 例 3: 画像生成モデル（Agent が画像を見て判断）

```
ユーザー: "Flux.1-dev を検証して"

Agent の動き:
1. black-forest-labs/FLUX.1-dev の model card を読む
2. 12GB+ VRAM → Colab Pro T4 では不足 → A100 or Modal
3. diffusers でロード、複数プロンプトで画像生成
4. 生成画像を見て（Claude のマルチモーダル能力）:
   - 「人物の手の描写が自然」
   - 「テキストレンダリングは弱い」
   - 「フォトリアリスティック系のプロンプトに強い」
5. 生成速度、ステップ数ごとの品質変化を測定
6. 画像付き HTML レポートを生成
```

## 固定コード vs Agent の使い分け

完全に Agent 任せではなく、**よく使う操作はユーティリティとして提供**する。
ただしそれは Agent が「使うことができるツール」であり、「必ず使うパイプライン」ではない。

```
agentic-bench/
├── src/
│   ├── utils/                    # Agent が使えるユーティリティ
│   │   ├── hf_model_info.py     # HF model card の情報取得
│   │   ├── gpu_estimator.py     # モデルサイズから VRAM 推定
│   │   ├── report_generator.py  # HTML レポートのテンプレート
│   │   └── metrics_writer.py    # metrics.json の書き出し
│   └── providers/               # プロバイダとのインターフェース
│       ├── colab_helpers.py     # Colab 操作のヘルパー
│       ├── modal_helpers.py     # Modal デプロイヘルパー
│       └── beam_helpers.py      # beam.cloud ヘルパー
├── skills/                      # Agent Skills (指示書)
│   ├── agentic-bench.md         # メインスキル
│   ├── beam-deploy.md           # beam.cloud 専用
│   └── colab-runner.md          # Colab 専用
└── results/                     # 検証結果
```

Agent は:
- utils/ のコードを**使っても使わなくても良い**
- 状況に応じて**新しいコードを書いても良い**
- providers/ のヘルパーは**プロバイダ固有の定型処理を楽にするためのもの**

---

## まとめ: なぜ Agent Skills が重要か

| 観点 | 固定パイプライン | Agent Skills |
|------|----------------|-------------|
| 新モデル対応 | コード追加が必要 | Agent が model card を読んで対応 |
| エラー対応 | 事前に想定した分のみ | Agent がデバッグ・リトライ |
| 品質判断 | 数値メトリクスのみ | Agent が出力を見て定性評価も |
| コスト最適化 | 固定ルール | Agent が状況判断 |
| レポート | テンプレート出力 | Agent が文脈に応じた解説を書く |
| 拡張性 | コード追加 | Skill の指示を更新 |
